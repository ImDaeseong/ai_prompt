## ⚠️ 절대 규칙 (프로젝트 유형 무관, 항상 적용)

아래 규칙은 어떤 프로젝트에서도 예외 없이 지킨다.

### 🔑 절대 코드에 넣지 않는 것
- API Key, Secret Key, Token, Password
- DB 접속 정보 (호스트, 포트, 계정, 비밀번호)
- 내부 서버 주소, 내부 네트워크 정보
- 개인 식별 정보 (이름, 이메일, 전화번호, 주민번호 등)
- 비공개 URL, 내부 문서 경로

**올바른 방법:**
```
# 잘못된 방법 ❌
API_KEY = "YOUR_API_KEY_HERE"

# 올바른 방법 ✅
import os
API_KEY = os.environ.get("API_KEY")
```

환경변수는 `.env` 파일로 관리하고, `.env`는 반드시 `.gitignore`에 추가한다.

### 🚫 코드 생성 시 AI에게 금지하는 것
- 하드코딩된 자격증명(credentials) 생성
- 기본 비밀번호(admin/1234/password) 사용
- 디버그 모드를 프로덕션 기본값으로 설정
- 모든 오리진 허용(`*`) CORS를 기본값으로 설정
- 에러 메시지에 스택 트레이스·내부 경로 노출
- 인증 없이 데이터를 수정·삭제할 수 있는 엔드포인트

## === [1단계] 프로젝트 유형 판별 ===

입력된 프로젝트 유형을 확인하고, 해당하는 섹션의 보안 요구사항을 전부 적용하라.
복합 프로젝트(예: 모바일 앱 + 백엔드 API)는 해당하는 모든 섹션을 합산 적용한다.

---

## === [2단계] 유형별 보안 요구사항 ===

### 🌐 A. 웹 서비스 (블로그, 관리자 페이지, SaaS 등)

#### 인증·인가
- [ ] 로그인이 있다면: 비밀번호는 반드시 bcrypt / argon2로 해시 저장 (평문 금지)
- [ ] 세션 토큰 또는 JWT는 HttpOnly 쿠키에 저장 (localStorage 저장 지양)
- [ ] JWT 사용 시: 만료 시간 설정 필수, 서명 알고리즘 HS256 이상
- [ ] 관리자 경로(`/admin`)는 일반 사용자와 분리된 인증 적용
- [ ] 비밀번호 재설정 링크는 일회성·만료 시간 포함

#### 입력·출력
- [ ] 모든 사용자 입력은 서버 측에서 유효성 검사 (프론트 검사만으로 불충분)
- [ ] DB 쿼리는 파라미터 바인딩 사용 (문자열 직접 조합 금지 → SQL Injection 방지)
- [ ] HTML 출력 시 이스케이프 처리 (`<script>` 등 → XSS 방지)
- [ ] 파일 업로드 시: 확장자·MIME 타입 검사, 실행 가능 경로에 저장 금지

#### 통신·서버
- [ ] HTTPS 강제 적용 (HTTP 접근은 HTTPS로 리다이렉트)
- [ ] CORS 허용 오리진은 명시적으로 제한 (`*` 사용 금지)
- [ ] CSRF 보호: POST/PUT/DELETE 요청에 CSRF 토큰 또는 SameSite 쿠키 적용
- [ ] 로그인 시도 횟수 제한 (Rate Limiting / 잠금 정책)
- [ ] 에러 페이지에서 프레임워크명·버전·서버 정보 노출 금지

#### 공개 불가 정보
- [ ] 서버 응답 헤더에서 `X-Powered-By`, `Server` 헤더 제거
- [ ] 소스코드 공개 시: `.env`, `config.py`, `settings.py` 내 실제 값 전부 제거 확인
- [ ] 로그 파일에 비밀번호·토큰이 기록되지 않도록 마스킹

---

### 📱 B. 모바일 앱 (Android, iOS, Flutter, React Native)

#### API Key·토큰 관리 (가장 중요)
- [ ] **API Key를 앱 소스코드·번들에 포함하지 않는다** (디컴파일로 추출 가능)
- [ ] 외부 API 호출은 반드시 자체 백엔드 서버를 경유하도록 설계
- [ ] 앱 내 저장이 불가피한 경우: Android Keystore / iOS Keychain 사용
- [ ] 환경별 빌드 설정으로 개발용·운영용 키를 분리

#### 통신
- [ ] 모든 통신은 HTTPS (HTTP 허용 설정 제거)
- [ ] 인증서 핀닝(Certificate Pinning) 검토 (민감 앱의 경우 필수)
- [ ] 요청·응답에 민감 정보(토큰·비밀번호)를 URL 파라미터로 전달하지 않음

#### 로컬 저장소
- [ ] SharedPreferences / AsyncStorage에 토큰·비밀번호 평문 저장 금지
- [ ] 민감 데이터는 암호화 후 저장 또는 Keystore/Keychain 사용
- [ ] 앱 로그(LogCat, Console)에 토큰·개인정보 출력 금지

#### 권한
- [ ] 사용하지 않는 권한 선언 제거 (카메라·위치·연락처 등)
- [ ] 권한 요청 시점은 실제 사용 직전으로 한정 (앱 시작 시 일괄 요청 금지)

#### 배포 전
- [ ] 디버그 로그·테스트 계정 제거 확인
- [ ] ProGuard / R8 난독화 적용 (Android)
- [ ] 앱 서명 키는 별도 안전한 장소에 보관 (소스코드 저장소에 포함 금지)

---

### 🖥️ C. 개인 소프트웨어 (데스크톱, CLI, 로컬 도구)

#### 로컬 전용이라도 지켜야 할 것
- [ ] 설정 파일(`config.json`, `settings.ini`)에 비밀번호·API Key 평문 저장 금지
  - 대안: OS 자격증명 저장소(Windows Credential Manager, macOS Keychain)
- [ ] 로컬 DB(SQLite 등)에 민감 정보 저장 시 암호화 검토
- [ ] 임시 파일·로그 파일에 민감 정보가 기록되지 않도록 처리

#### 외부 통신이 있는 경우
- [ ] HTTPS만 사용, 인증서 검증 비활성화 코드 금지
- [ ] API Key는 소스코드 외부(환경변수·설정 파일)에서 주입
- [ ] 업데이트 기능이 있다면: 업데이트 파일 무결성 검증(서명) 적용

#### 배포·공개 시
- [ ] 소스코드 공개 전: `git log`·`git diff` 전체 검토 (과거 커밋에 secrets 포함 여부)
- [ ] `git-secrets` 또는 `truffleHog` 등으로 히스토리 스캔 권장
- [ ] 실행 파일 배포 시: 테스트용 하드코딩 값 전량 제거 확인

## === [3단계] API 설계 보안 요구사항 ===

외부 API를 사용하거나 자체 API를 제공하는 경우 적용한다.

### 외부 API 사용 시
- [ ] API Key는 서버 측에서만 사용 (클라이언트·프론트엔드에 노출 금지)
- [ ] API Key는 환경변수로 주입, 소스코드 저장소에 커밋 금지
- [ ] 사용하는 외부 서비스의 최소 권한 원칙 적용 (필요한 스코프만 요청)
- [ ] 외부 API 응답을 그대로 클라이언트에 전달하지 않음 (필요한 필드만 선별 전달)

### 자체 API 제공 시
- [ ] 모든 엔드포인트에 인증 적용 (공개 API가 아닌 이상 인증 없는 접근 차단)
- [ ] 입력 값 유효성 검사를 API 계층에서 수행
- [ ] 에러 응답에 내부 구조(테이블명, 파일 경로, 스택 트레이스) 포함 금지
- [ ] Rate Limiting 적용 (엔드포인트별 초당/분당 요청 수 제한)
- [ ] API 버전 관리 (`/api/v1/`) 설계 시점에 결정

## === [4단계] 공개 저장소 체크리스트 ===

GitHub 등에 올리기 전 반드시 확인한다.

```
확인 명령 (secrets 스캔):
git log --all --full-history -- "*.env"
git grep -i "password\|api_key\|secret\|token" $(git log --pretty=format:%H)
```

- [ ] `.gitignore` 에 포함 여부 확인
  ```
  .env
  .env.local
  .env.production
  config/secrets.*
  *.key
  *.pem
  ```
- [ ] README에 설정 방법은 기술하되 실제 값은 절대 포함하지 않음
- [ ] 샘플 설정 파일은 `.env.example` 형태로 제공 (실제 값 없이 키 이름만)
