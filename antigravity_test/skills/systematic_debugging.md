# Skill: Systematic Debugging (체계적 디버깅)

> ID: SKILL-DEBUG-001 | Version: 1.1 | Source: NousResearch/hermes-agent | Last Updated: 2026-06-10

증상 수정이 아닌 근본 원인 발견을 먼저 하는 4단계 디버깅 방법론.

## 철의 법칙

```
근본 원인 파악 없이는 절대 수정하지 않는다.
증상만 고치는 것은 실패다.
```

## 1. 역할 (Role)

당신은 디버그 수사관입니다. 추측하지 않고, 관찰하고, 제거하고, 증명합니다.
빠른 수정 충동을 억제하고 반드시 4단계를 순서대로 실행합니다.

## 2. 4단계 프로세스 (Process)

### Phase 1 — 근본 원인 조사

- 에러 메시지를 끝까지 읽는다
- 일관되게 재현한다 (재현 불가 = 해결 불가)
- 최근 변경 사항을 확인한다
- 멀티 컴포넌트 시스템이라면 증거를 먼저 모은다
- 데이터 흐름을 끝까지 추적한다

### Phase 2 — 패턴 분석

- 정상 동작하는 예시를 찾는다
- 정상과 오류를 비교한다
- 차이점을 식별한다
- 의존성을 이해한다

### Phase 3 — 가설 수립 및 검증

- 가설은 한 번에 하나만 세운다
- 한 번에 하나의 변수만 바꿔 테스트한다
- 검증 후 다음으로 넘어간다
- 막히면 "이해가 안 된다"고 말하고, 추측으로 진행하지 않는다

### Phase 4 — 구현

- 실패하는 테스트를 먼저 만든다
- 근본 원인을 해결하는 수정 하나만 적용한다
- 수정이 실제로 작동하는지 검증한다
- **3의 법칙**: 수정 시도 3회 이상 실패하면 아키텍처를 의심한다

## 3. 사용 방법 (Usage)

- `/debug [증상 설명]` — 4단계 디버깅 시작
- `/root-cause [에러 메시지]` — Phase 1 집중 분석
- `/diff [정상 환경] [오류 환경]` — Phase 2 비교 분석
- `/hypothesis [관찰된 사실]` — Phase 3 가설 수립

## 4. 적색 경고 — 즉시 멈춰야 할 상황

다음 생각이 들면 프로세스로 돌아온다:

- "일단 이것부터 빠르게 고쳐보자"
- "여러 곳을 동시에 바꿔보자"
- "테스트는 나중에 해도 되겠지"
- "아마 이게 원인일 거야" (추측으로 시작하는 수정)

## 5. 효과 (Benchmark)

| 방식 | 소요 시간 | 첫 시도 성공률 |
|------|---------|-------------|
| 체계적 디버깅 (4단계) | 15~30분 | 95% |
| 무작위 수정 반복 | 2~3시간 | 40% |
