# Skill: Agent Architecture Audit (에이전트 아키텍처 진단)

> ID: SKILL-AGENT-AUDIT-001 | Version: 1.0 | Source: affaan-m/ECC | Last Updated: 2026-06-11

LLM/에이전트 앱의 12레이어 스택을 진단해 래퍼 회귀, 메모리 오염, 도구 실패, 숨겨진 루프, 렌더링 손상을 찾아낸다.

**Iron Law**: 모델을 탓하기 전에 먼저 래퍼 레이어 회귀를 반증하라.

---

## 활성화 조건

**필수 사용:**
- 에이전트/LLM 앱을 프로덕션에 배포하기 전
- 도구 호출·메모리·다단계 워크플로우 포함 기능 출시 전
- "에이전트가 점점 나빠진다" "도구가 불안정하다" 보고 시
- 플레이그라운드에서는 작동하는데 래퍼 안에서 깨질 때
- 원인 없이 15분 이상 디버깅 중일 때

---

## 12레이어 스택

모든 에이전트 시스템에는 이 레이어가 있다. 어느 레이어에서나 오답이 나올 수 있다.

| # | 레이어 | 주요 장애 |
|----|-------|----------|
| 1 | 시스템 프롬프트 | 지시 충돌, 프롬프트 비대화 |
| 2 | 세션 히스토리 | 이전 턴의 낡은 컨텍스트 주입 |
| 3 | 장기 메모리 | 세션 간 오염, 과거 주제 유입 |
| 4 | 증류(Distillation) | 압축 아티팩트가 유사사실로 재진입 |
| 5 | 능동 회상 | 중복 재요약 레이어로 컨텍스트 낭비 |
| 6 | 도구 선택 | 잘못된 라우팅, 필수 도구 건너뜀 |
| 7 | 도구 실행 | 호출했다고 주장하지만 실제로는 미실행 |
| 8 | 도구 해석 | 도구 출력을 오독하거나 무시 |
| 9 | 응답 형성 | 최종 응답에서 포맷 손상 |
| 10 | 플랫폼 렌더링 | UI/API/CLI가 유효한 응답을 변이 |
| 11 | 숨겨진 수리 루프 | 두 번째 LLM 패스를 조용히 실행 |
| 12 | 영속화 | 만료된 상태나 캐시 아티팩트를 현행 증거로 재사용 |

## 주요 실패 패턴 5가지

### 1. 래퍼 회귀
기본 모델은 정답을 내지만 래퍼 레이어가 악화시킨다.
- 증상: 직접 API 호출은 OK, 에이전트 안에서 깨짐 / 새 프롬프트 레이어 추가 후 기존 동작 저하

### 2. 메모리 오염
히스토리·메모리 검색·증류를 통해 과거 주제가 새 대화에 누출된다.
- 증상: 무관한 과거 주제 언급 / 사용자 수정이 적용 안 됨 / 메모리 무제한 증가

### 3. 도구 규율 실패
도구가 프롬프트에만 선언되고 코드에서 강제되지 않는다.
- 증상: "반드시 도구 X 사용"인데 호출 없이 답변 / 도구 결과가 실제로는 실행 안 됨

### 4. 렌더링/전송 손상
에이전트 내부 응답은 정확한데 플랫폼이 전달 중에 변이시킨다.
- 증상: 로그에는 정답, 사용자에게는 깨진 출력 / 스트리밍 파편화

### 5. 숨겨진 에이전트 레이어
조용한 수리·재시도·요약 에이전트가 계약 없이 실행된다.
- 증상: 내부 생성과 사용자 전달 사이에 출력이 달라짐 / "자동 수정" 루프가 응답을 교체

---

## 진단 워크플로우

**Phase 1 — 범위 정의**: 대상 시스템, 진입점, 모델 스택, 증상, 시간 범위, 감사할 레이어  
**Phase 2 — 증거 수집**: 소스 코드(에이전트 루프·도구 라우터·메모리 승인·프롬프트 조립), 로그, 설정  
**Phase 3 — 실패 매핑**: 증상 → 메커니즘 → 소스 레이어 → 근본 원인 → 증거 참조 → 신뢰도  
**Phase 4 — 수정 전략** (코드 우선, 프롬프트 우선 금지):

1. 도구 요구사항을 코드로 게이팅 (프롬프트 텍스트만으로 강제 금지)
2. 숨겨진 수리 에이전트 제거 또는 명시적 계약으로 교체
3. 컨텍스트 중복 제거 (시스템 프롬프트 + 메모리 + 히스토리에 같은 정보 반복 금지)
4. 메모리 승인 강화 (사용자 수정 > 에이전트 주장)
5. 렌더링 변이 최소화 (pass-through, 변환 금지)

## 심각도 모델

| 레벨 | 의미 | 조치 |
|------|------|------|
| critical | 에이전트가 자신 있게 잘못된 운영 동작 생성 | 다음 배포 전 수정 |
| high | 정확도·안정성이 자주 저하 | 이번 스프린트 수정 |
| medium | 정확도는 대체로 유지되지만 출력이 취약하거나 낭비적 | 다음 사이클 계획 |
| low | 주로 코드 유지보수 이슈 | 백로그 |

## 빠른 진단 체크리스트

| # | 질문 | YES이면 → |
|----|------|----------|
| 1 | 모델이 필수 도구 없이도 답변할 수 있는가? | 도구가 코드로 게이팅되지 않음 |
| 2 | 과거 대화 내용이 새 턴에 나타나는가? | 메모리 오염 |
| 3 | 같은 정보가 시스템 프롬프트·메모리·히스토리에 모두 있는가? | 컨텍스트 중복 |
| 4 | 전달 전에 플랫폼이 두 번째 LLM 패스를 실행하는가? | 숨겨진 수리 루프 |
| 5 | 내부 생성과 사용자 전달 출력이 다른가? | 렌더링 손상 |
| 6 | "반드시 도구 X 사용" 규칙이 프롬프트 텍스트에만 있는가? | 도구 규율 실패 |

## 피해야 할 안티패턴

- 래퍼 레이어 회귀를 반증하기 전에 모델을 탓하지 않는다
- 오염 경로를 보여주지 않고 메모리를 탓하지 않는다
- 현재 상태가 깨끗하다고 과거 오염 이력을 지우지 않는다
- 마크다운 프로즈를 신뢰할 수 있는 내부 프로토콜로 취급하지 않는다
- 코드가 강제하지 않는데 "반드시 도구 사용"을 프롬프트에만 쓰지 않는다
