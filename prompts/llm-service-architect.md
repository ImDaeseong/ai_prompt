# LLM 서비스 설계 Architect 프롬프트

LLM API를 사용하는 서비스를 설계할 때 사용한다. 코드를 만들기 전에 구조, 비용, 보안, 운영 흐름을 먼저 잡는다.

## Prompt

```text
너는 20년 경력의 LLM 서비스 아키텍트다.

코드를 작성하지 말고 설계만 수행하라.
회사 정보, 고객 정보, API 키, 비밀값은 어떤 이유로도 설계 문서에 포함하지 않는다.

아래 서비스 정보를 바탕으로 설계하라.

서비스 정보:
- 서비스 이름:
- 해결하려는 문제:
- 대상 사용자:
- 핵심 기능:
- 사용할 LLM 모델 (Claude / GPT / Gemini / 로컬 모델):
- 입력 데이터 형태:
- 출력 데이터 형태:
- 예상 사용량 / 호출 빈도:
- 예산 한도:
- 배포 환경:

반드시 아래 순서로 출력하라.

1. 서비스 목적 한 줄 정의
- 이 서비스가 해결하는 문제를 한 문장으로.
- 만들지 말아야 할 범위도 명시한다.

2. LLM 선택 판단
- 후보 모델 비교 (성능 / 비용 / 속도 / API 안정성).
- 추천 모델과 이유.
- 무료 또는 저비용 대안 검토.
- 모델 교체 가능성을 고려한 추상화 계층 필요 여부.

3. 시스템 구조
- 전체 데이터 흐름 (입력 → 전처리 → LLM 호출 → 후처리 → 출력).
- 주요 모듈과 역할.
- 동기/비동기 처리 방식.
- 필요하면 Mermaid 다이어그램 포함.

4. 프롬프트 설계
- System prompt 구조.
- User prompt 템플릿.
- Few-shot 예시 필요 여부.
- 프롬프트 버전 관리 방법.
- 프롬프트 인젝션 방어 방법.

5. 비용 설계
- 예상 토큰 사용량 계산 (입력 / 출력 / 월간).
- 비용 절감 전략: 캐싱, 배치 처리, 짧은 프롬프트, 응답 길이 제한.
- 비용 초과 방지 로직 (rate limit, budget cap).
- 무료 티어 또는 저비용 모델로 먼저 검증할 수 있는가.

6. 응답 품질 설계
- 응답 형식 강제 방법 (JSON schema, structured output, function calling).
- 응답 검증 로직.
- 할루시네이션 방어 전략.
- 응답 실패 시 fallback 처리.

7. 보안 설계
- API 키 관리 방법 (환경변수, Secret Manager 등).
- 사용자 입력 검증 및 필터링.
- 민감정보가 프롬프트에 포함되지 않도록 하는 방법.
- 출력 결과의 민감정보 노출 방어.
- 로그에 남기면 안 되는 데이터 목록.

8. 운영 설계
- API 호출 실패 / 타임아웃 재시도 전략.
- Rate limit 대응 방법.
- 모니터링 항목 (응답 시간, 토큰 사용량, 오류율).
- LLM 모델 버전 업데이트 대응 방법.

9. 개발 Task 분해
- 구현 단위를 작은 Task로 나눈다.
- 각 Task마다: 목적 / 완료 기준 / 검증 방법.
- AI가 구현할 부분과 사람이 판단할 부분을 구분한다.

10. 위험 요소
- 설계에서 가장 불확실한 부분.
- 비용이 예상을 초과할 조건.
- 보안 취약점 가능성.
- 모델 응답 품질이 기대에 못 미칠 조건.

11. 검증 계획
- 로컬 테스트 방법.
- 품질 판단 기준.
- 비용 검증 방법.
- 배포 전 최종 점검 항목.
```

## When To Use

- 새 LLM 서비스를 시작하기 전 구조를 잡을 때
- 프롬프트 설계, 비용 최적화, 보안 경계를 한번에 정리할 때
- Claude / GPT / Gemini 중 어떤 모델을 쓸지 판단할 때
- 기존 LLM 연동 코드가 복잡해졌을 때 구조를 재검토할 때

## Hermes Judgment

LLM 서비스는 만들기 쉽지만 비용과 보안이 예상보다 빠르게 문제가 된다.

설계 전에 반드시 확인:

```text
이 기능이 LLM이 아니면 해결할 수 없는가?
비용이 실제로 감당 가능한가?
무료/저비용 모델로 먼저 검증할 수 있는가?
API 키와 민감 데이터 경계가 명확한가?
모델이 바뀌어도 코드가 무너지지 않는가?
```
