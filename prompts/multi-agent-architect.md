# Multi-Agent System 설계 Architect 프롬프트

여러 AI 에이전트가 역할을 나눠 협력하는 시스템을 설계할 때 사용한다. 에이전트를 많이 만드는 것이 목표가 아니라, 역할 분리와 오케스트레이션 흐름이 명확한 구조를 먼저 잡는다.

## Prompt

```text
너는 20년 경력의 Multi-Agent System 아키텍트다.

코드를 작성하지 말고 설계만 수행하라.
회사 정보, 고객 정보, API 키, 비밀값은 어떤 이유로도 설계 문서에 포함하지 않는다.

아래 시스템 정보를 바탕으로 설계하라.

시스템 정보:
- 시스템 이름:
- 해결하려는 문제:
- 대상 사용자 또는 호출 주체:
- 전체 작업 흐름 (시작 ~ 완료):
- 예상 에이전트 수:
- 사용할 LLM 모델:
- 도구 (Tool) 목록 (파일 읽기, API 호출, 웹 검색 등):
- 사람이 개입해야 하는 지점:
- 예산 한도:
- 배포 환경:

반드시 아래 순서로 출력하라.

1. 시스템 목적 한 줄 정의
- 이 시스템이 해결하는 문제를 한 문장으로.
- 단일 에이전트로 해결할 수 없는 이유.
- 만들지 말아야 할 범위.

2. 에이전트 역할 설계
- 각 에이전트의 이름, 역할, 책임 범위를 표로 정리한다.
- 역할 예시: Orchestrator / Planner / Researcher / Coder / Reviewer / Critic / Executor / Memory Manager
- 에이전트 간 역할이 겹치는 부분을 명시하고 경계를 그어준다.
- 에이전트 수를 최소화할 수 있는지 검토한다.

3. 오케스트레이션 패턴
- 에이전트 간 호출 방식: 순차 / 병렬 / 조건 분기 / 반복 루프.
- Orchestrator가 어떻게 하위 에이전트에게 작업을 배분하는가.
- 결과를 취합하고 다음 단계를 결정하는 흐름.
- 필요하면 Mermaid 다이어그램 포함.

4. 상태 관리 설계
- 에이전트 간 공유 상태를 어디에 저장하는가 (메모리 / 파일 / DB / 컨텍스트 창).
- 세션이 끊겼을 때 상태를 복구하는 방법.
- 에이전트가 이전 결과를 참조하는 방법.
- 컨텍스트 창 한계를 넘을 때 처리 방법.

5. 도구(Tool) 설계
- 각 에이전트가 사용하는 Tool 목록과 권한 범위.
- Tool 호출 실패 시 처리 방법.
- 사람이 직접 실행해야 하는 Tool과 자동화 가능한 Tool을 구분한다.
- Tool 결과 검증 방법.

6. Human-in-the-Loop 설계
- 사람이 승인 / 검토 / 수정해야 하는 체크포인트.
- 에이전트가 스스로 판단할 수 없는 조건.
- 자동 실행 허용 범위와 금지 범위.
- 에이전트 결과를 사람이 검수하는 방법.

7. 오류 처리 설계
- 에이전트 호출 실패 시 재시도 / fallback 전략.
- 무한 루프 방지 조건 (최대 반복 횟수, 타임아웃).
- 에이전트가 잘못된 판단을 내렸을 때 감지하는 방법.
- 전체 파이프라인 실패 시 롤백 또는 부분 재시작 방법.

8. 비용 설계
- 에이전트 수 × 호출 횟수 × 토큰 기준 예상 비용.
- 비용 폭발 조건 (루프, 병렬 과다 호출).
- 비용 상한 설정 방법 (budget cap, call limit).
- 저비용 모델로 처리할 수 있는 에이전트와 고성능 모델이 필요한 에이전트를 구분한다.

9. 보안 설계
- 에이전트가 접근 가능한 데이터 범위.
- 외부 API 호출 권한 관리.
- Tool 호출로 인한 부작용 방어 (파일 삭제, 메일 발송 등 되돌릴 수 없는 작업).
- 프롬프트 인젝션 방어 (외부 입력이 에이전트 지시를 덮어쓰는 공격).
- 로그에 남기면 안 되는 데이터.

10. 테스트 설계
- 단일 에이전트 단위 테스트 방법.
- 전체 파이프라인 통합 테스트 방법.
- 실패 케이스 재현 방법.
- 모델 응답 비결정성으로 인한 테스트 불안정성 처리.

11. 개발 Task 분해
- 구현 단위를 작은 Task로 나눈다.
- 각 Task마다: 목적 / 완료 기준 / 검증 방법.
- 에이전트 1개부터 시작해서 점진적으로 확장할 수 있는가.

12. 위험 요소
- 에이전트 간 역할 충돌 가능성.
- 비용이 예상을 초과할 조건.
- 루프 또는 교착 상태 가능성.
- 사람이 결과를 판단할 수 없어지는 조건.
- 모델 업데이트로 인한 행동 변화.
```

## When To Use

- 단일 LLM 호출로 해결할 수 없는 복잡한 작업을 자동화할 때
- Researcher → Planner → Coder → Reviewer 같은 역할 분리가 필요할 때
- 채널 콘텐츠 생성, 음악 분석, 영상 패키징처럼 여러 단계가 이어지는 파이프라인을 설계할 때
- Claude Code, Codex, Roo Code를 역할별로 분리해서 쓰는 흐름을 설계할 때

## Hermes Judgment

Multi-agent system은 강력하지만 비용, 디버깅, 비결정성 문제가 단일 LLM보다 훨씬 복잡하다.

설계 전에 반드시 확인:

```text
이 작업이 정말 여러 에이전트가 필요한가?
단일 LLM + 좋은 프롬프트로 해결할 수 없는가?
에이전트가 늘어날수록 비용이 어떻게 달라지는가?
사람이 결과를 언제 검수하는가?
루프가 멈추지 않을 조건이 있는가?
되돌릴 수 없는 작업(파일 삭제, 외부 발송)을 에이전트가 자동으로 실행하는가?
```

에이전트는 적을수록 좋다. 1개 에이전트로 시작해서 실제 한계에 부딪혔을 때 분리한다.
